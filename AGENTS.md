# AGENTS.md

## 核心理念

- 以用户目标为中心，不以功能堆砌为中心。
- 用户体验优先于技术偏好、代码整洁度和架构优雅度。
- 优先清晰、一致、可预期，避免炫技式设计。
- 优先解决真实问题，不追求形式上的“正确”。
- 不把局部实现偏好上升为全局原则。
- 选择适合当前问题的简单方案。

## 交互设计

- 用户体验优先于技术偏好、代码整洁度和架构优雅度。
- 后端可以复杂，但用户触碰到的每一层都必须简单、顺滑、可预期。
- 交互体验不只包含 GUI，也包含 CLI、对话式交互、Skill、系统反馈和自动化流程。
- 为目标设计，不为功能设计。
- 不要让用户思考。
- 能推断就不询问。
- 能自动化就不手动化。
- 能一步完成就不要拆成多步。
- 优先提供合理默认值，减少强制配置。
- 采用渐进式展示。
- 优先展示核心路径，高级能力按需展开。
- 反馈不仅要说明问题，还要指导下一步。
- 减少打扰，但保留用户掌控感。
- 同类问题尽量保持同类交互方式。
- 不仅要缩短成功路径，也要设计清楚失败后的恢复路径。

## 系统设计

- 遵循 DRY、KISS、YAGNI。
- 保持低耦合、清晰边界、显式依赖。
- 优先可理解性，不追求技巧性。
- 结构应服务真实问题，不为假设性未来提前设计。
- 先解决当前真实需求，再考虑抽象和扩展。
- 默认行为应安全、稳妥、可回退。
- 重要状态、依赖和副作用必须可见。
- 一致性优先于局部最优。

## 实现原则

- 不预设某种编程范式优先。
- 选择最适合问题、团队和项目上下文的实现方式。
- 优先复用，避免重复实现。
- 模块保持聚焦，职责尽量单一。
- 避免隐藏输入、隐藏状态、隐藏副作用。
- 函数和模块应尽量保持单一职责，避免通过 flag 或分支参数混合多种行为。
- 输入、输出、失败方式应尽量显式。
- 命名应表达真实语义，避免模糊命名。
- 类型系统允许时，优先强类型和清晰建模。
- 注释用于补充上下文、约束和原因，不用于解释显而易见的代码，也不用于掩盖糟糕设计。

## 错误处理

- 失败必须显式表达，禁止静默失败。
- 抛出具体错误，禁止吞错。
- 禁止用 catch-all 掩盖根因。
- 外部依赖默认都可能失败。
- 外部调用应重试、告警，最终失败要抛错。
- 错误反馈应可定位、可行动、可恢复。
- 错误上下文应包含输入参数、状态码、响应内容、依赖信息。
- 日志优先结构化，避免把动态信息直接拼进字符串。

## 依赖与工具

- 使用项目级依赖管理。
- 不依赖全局安装。
- 依赖写入项目配置，不做一次性手工安装。
- 行为不确定时，先读源码，再做判断。
- 优先稳定、成熟、可维护的工具。

## 测试

- 尊重项目现有测试策略。
- 不默认新增测试。
- 优先集成测试、E2E、冒烟测试。
- 单元测试只在确有必要时使用。
- 能验证真实行为，就不要过度依赖 Mock。
- 测试目标是验证真实行为，不是追求覆盖率数字。
- 只补最小且有效的测试覆盖。

## 工作流

- 修改前先理解现有代码、上下文和约束。
- 修改前先读当前生效的 `AGENTS.md`。
- 先搜索已有实现，再新增逻辑。
- 改动保持最小化，聚焦当前问题。
- 不回退无关修改。
- 保持与项目既有风格一致。
- 优先使用非交互式命令。
- 改动后执行现有构建、lint、测试或必要验证。

## 文档

- 代码是第一文档。
- 优先用命名、结构、类型表达语义。
- 需要时补充 docstring。
- 避免重复文档。
- 文档描述当前事实，不写变更流水账。

## 硬规则

- 注释仅使用英文。
- import 放文件顶部。
- 不修改输入参数，除非语言或框架惯例明确要求。
- 关键参数显式传入；默认参数只用于稳定、直观、非关键的可选项。
- 不写多模式函数，不写 flag-driven behavior。
- 非必要不新增抽象层。
- 非必要不新增依赖。
- 非必要不新增测试。
